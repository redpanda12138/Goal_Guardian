# 语音转录失败原因分析

## 一、录音格式 vs 模型要求

### 1. 客户端录音格式（talkieai-uniapp）

| 运行环境 | 格式 | 采样率 | 说明 |
|----------|------|--------|------|
| **H5** | WAV（recorder-core） | **32 000 Hz** | 单声道，比特率 32 |
| **微信/App** | WAV 或 **MP3（Android）** | **44 100 Hz** | RecorderManager，Android 上多为 mp3 |

### 2. Whisper 模型要求（HF / 本地 Sing2Eng）

- **输入**：16 000 Hz、单声道、float 波形（由 Processor 转成 log-Mel 特征）。
- **文档**：Whisper 标准采样率为 16 kHz。

### 3. 服务端处理（格式已对齐）

- **上传**：`save_voice_file` 按二进制原样保存；若为 `.mp3`（如 Android），`sys_service.voice_upload` 会用 pydub 转为 WAV 并返回新的 `.wav` 文件名。
- **校验**：`validate_audio_file` 用 `librosa.load(audio_path, duration=1.0)` 试读 1 秒，失败则返回 400「音频文件无效或损坏」。
- **转录前**：`whisper_voice.preprocess_audio` 用 `librosa.load(sr=None, mono=True)` 读取，再 `librosa.resample` 到 **16 000 Hz**，并用 soundfile 写出临时 WAV。
- **推理**：`transcribe_audio` 内再次 `librosa.load(path_to_use, sr=16000, mono=True)`，得到 16 kHz 单声道 float 后交给 Processor。

**结论：采样率、声道、格式在服务端都会被统一成 16 kHz 单声道再送入模型，因此「录音格式与模型要求不一致」不是当前转录失败的主要原因。**

## 二、转录失败/结果为空的可能原因

### 1. 模型与任务不匹配（主因）

- 当前使用 **ivabojic/whisper-small-sing2eng-translate**，面向 **Singlish→英文翻译**。
- 若录音为**标准英语、非 Singlish、或非英语**，模型容易产生：
  - 退化输出（如 `!!A!!B!!C`、长串标点）；
  - 或有效字符占比很低。
- 服务端 `_normalize_transcription` 会把这些结果判为无效并返回空，前端表现为「未识别到内容」或「No speech detected」。

**已做**：当 `task="translate"` 两次均未得到有效结果时，自动再试一次 `task="transcribe"`（标准英语听写），便于标准英语录音出字。

### 2. 音频质量与时长

- 日志中已有：`duration < 0.3s`、`peak≈0`（几乎静音）时打 warning。
- 过短、过小声或环境噪声大，模型也容易输出占位符或无效内容，被归一化丢弃。

**已做**：在送入模型前增加**可选降噪**（安装 `noisereduce` 后自动启用），噪声较大时可提升识别率。安装：`pip install noisereduce`。

### 3. 文件与校验

- **文件不存在**：返回 404。
- **校验失败**：`validate_audio_file` 抛错（如损坏、librosa 不支持的编码）→ 400「音频文件无效或损坏」。
- **上传为空**：`voice_upload` 中若 `file_size == 0` 会抛异常。

### 4. 模型未初始化

- 若 Whisper 加载失败，`whisper_processor.pipe` 与 processor/model 均为 None，接口会返回「语音识别模型未初始化」。
- 接口层应使用与 `transcribe_audio` 相同的判断：`(processor and model) or pipe`，避免只检查 `pipe` 导致误判。

---

## 三、建议排查顺序

1. **看 nohup.out 日志**  
   - 是否有「Whisper 返回空或无效」「语音转录结果为空」→ 多为模型输出被归一化丢弃或静音/过短。  
   - 是否有「音频过短」「几乎静音」→ 改善录音时长与音量。  
   - 是否有「音频预处理失败」「音频文件无效」→ 检查上传文件格式与 `validate_audio_file`。

2. **确认实际使用的文件**  
   - 语音转文字用的是 `voice-translate` 里 `voice_file_get_path(dto.file_name)` 得到的路径。  
   - 若上传为 mp3，返回的已是 `.wav` 文件名，需用该 wav 路径去转录。

3. **确认模型与任务**  
   - 当前为 Sing2Eng、`task="translate"`；若需「听写」或通用多语言，需换模型或改 task。

4. **可选：客户端采样率**  
   - 客户端 32 kHz / 44.1 kHz 均可，服务端会重采样到 16 kHz。若希望减少服务端负载，可在 H5 将录音采样率改为 16 kHz（需确认 recorder-core 支持），非必须。

## 四、小结

| 项目 | 是否一致 | 说明 |
|------|----------|------|
| 采样率 | ✅ 服务端统一 | 任意采样率 → 预处理为 16 kHz |
| 声道 | ✅ 服务端统一 | librosa mono=True → 单声道 |
| 格式 | ✅ 兼容 | WAV/MP3 均可，MP3 会先转 WAV |
| 模型任务 | ⚠️ 易导致“无结果” | Sing2Eng 针对 Singlish，非 Singlish 易被归一化丢弃 |

**转录“不成功”主要是模型输出被判定为无效（任务/语言/质量），而不是录音文件格式与模型要求不一致。**

## 五、根据 nohup 日志的失败原因分析（示例 31734–31748）

**现象**：  
- 音频正常：5.29s，peak=0.4479，已预处理为 16kHz。  
- 三次尝试均执行：第一次（README 式）→ 第二次（forced_decoder_ids translate）→ 第三次（task=transcribe）。  
- 最终仍为「返回空或无效」。

**结论**：  
三种解码方式（默认 config、显式 translate、显式 transcribe）得到的**原始输出**都被 `_normalize_transcription` 判为无效（空、仅占位符、或退化模式如 !!A!!B!!C / 大量标点 / 有效字符占比过低等）。

**可能原因**：  
1. **模型与输入不匹配**：当前为 **Sing2Eng**（[Singlish2English](https://github.com/IvaBojic/Singlish2English)），针对 Singlish/新加坡口音微调。若录音是**标准英语、非 Singlish、或无新加坡口音**，属于分布外输入，模型容易输出占位符或退化结果。  
2. **归一化过严**：若模型实际输出了少量有效词但混在大量标点/重复中，会被「有效字符占比 < 40%」或「!! 过多」等规则整段丢弃。

**建议**：  
1. **看“末次原始输出”**：失败时现已打日志 `Whisper 末次原始输出(供排查): len=..., sample=...`。下次失败后查看 nohup.out 中该行，确认模型到底返回的是空、纯标点、还是带少量英文的退化串。  
2. **若主要为标准英语**：考虑改用通用 Whisper（如 `openai/whisper-small` 或 `whisper-large-v3`）并设 `task="transcribe"`，或增加配置在「Sing2Eng」与「通用 Whisper」间切换。  
3. **若必须用 Sing2Eng**：尽量在 Singlish/新加坡口音或与训练分布接近的录音上使用；标准英语清晰录音可能持续得不到有效结果。

## 六、根本原因排查与修正（当前策略）

- **已移除**：通用 Whisper（openai/whisper-small）回退调用，不再在 Sing2Eng 失败后尝试在线/本地通用模型。
- **诊断**：每次转录会打 `Whisper input_features: shape=..., min=..., max=..., mean=...`，用于确认送入编码器的特征是否正常；若 shape 异常或 min/max/mean 极端，可能是预处理或设备导致退化。
- **防退化**：  
  - `suppress_tokens` 在三次 `generate` 中均传入，包含 config 原有列表加上 **"!" "$" "#" "%" "&"** 的 token id，避免 decoder 反复输出同一符号（先出现 444 个 `!`，抑制后变为 444 个 `$`，故一并抑制常见符号）。  
  - 第一次生成使用 **num_beams=5** 与 **repetition_penalty=1.2**（不再用 no_repeat_ngram_size=3，否则易导致只输出 1 个字符如 `2`）。  
  - 若仍出现 444 个单引号 `'` 或逗号 `,` 等退化，已把 **`'` `"` `,`** 等加入 suppress_tokens（会逐次根据日志补充）。

**根本原因**：Sing2Eng 在**非 Singlish（如标准英语）** 输入上易发生 **decoder 单 token 重复**，抑制一批符号后仍会换用下一符号（如 `,` `.` `-`）。彻底解决需二选一：**对标准英语使用通用 Whisper（如 openai/whisper-small）**，或 **仅对 Singlish/新加坡口音使用 Sing2Eng**。

## 七、通用 Whisper 回退（可选，已不默认使用）

当 Sing2Eng 三次均无效（如全 `!`）时，可启用**通用 Whisper 回退**，用 `openai/whisper-small` 再做一次 `task=transcribe`，适合标准英语等非 Singlish 录音。

**启用方式**：设置环境变量后重启服务。

- Linux / macOS（当前 shell）：  
  `export WHISPER_FALLBACK_TO_GENERIC=1`
- Windows CMD：  
  `set WHISPER_FALLBACK_TO_GENERIC=1`
- Windows PowerShell：  
  `$env:WHISPER_FALLBACK_TO_GENERIC="1"`
- 或在启动命令前加：  
  `WHISPER_FALLBACK_TO_GENERIC=1 nohup python ... &`

**行为**：首次在 Sing2Eng 三次都失败时，会尝试加载通用模型；加载成功后同进程内复用。日志会出现 `Whisper 使用通用模型回退(openai/whisper-small)` 与 `Whisper 通用模型回退成功，长度: ...`。

**离线 / 无网络**：服务器无法访问 Hugging Face 时，需使用本地通用模型：

1. **在能联网的机器上下载**（一次性）：
   ```bash
   pip install huggingface_hub
   huggingface-cli download openai/whisper-small --local-dir ./whisper-generic-offline
   ```
   将生成的 `whisper-generic-offline` 目录拷贝到服务器的 `talkieai-server/mas/` 下（即 `mas/whisper-generic-offline`）。

2. **或指定自定义路径**：设置环境变量  
   `WHISPER_GENERIC_MODEL_PATH=/绝对路径/whisper-generic-offline`，代码会优先从该路径加载，不再访问网络。

未设置 `WHISPER_GENERIC_MODEL_PATH` 时，默认会先查找 `mas/whisper-generic-offline`，存在则从本地加载；不存在再尝试在线拉取。
